Getting Started with the datapill CLI

Last updated: May 2, 2026  |  Version: 0.2.9 beta

datapill is a command-line tool for ingesting, inspecting, and managing data pipelines. It connects to databases, object storage, message queues, and REST APIs - streaming data into tracked artifacts you can query, compare, and version.

ℹ️ What you'll learn
By the end of this guide you will be able to install datapill, connect to a data source, run your first ingestion, and inspect the resulting artifact from the command line.

Core concepts

  • Source - A named connector type (postgres, s3, kafka, etc.)
  • Config file - A .json file holding connection parameters for a source.
  • Artifact - A versioned record of a pipeline run, optionally materialized as Parquet on disk.
  • Store - A local .datapill/ directory that tracks all artifacts via DuckDB.

Installation

datapill requires Python 3.11 or newer.

Install from PyPI

bash
$ pip install datapill

Verify the installation

bash
$ dp --version
datapill 0.2.9 beta

$ dp --help
Usage: dp [OPTIONS] COMMAND [ARGS]...

  datapill - lightweight data ingestion and pipeline toolkit.

Commands:
  artifact  inspect and manage pipeline artifacts
  ingest    ingest data from a source into datapill
  profile   profile a dataset artifact

Quick Start

The following steps walk you through ingesting a local CSV/Parquet file, profiling it, and inspecting the results — no database required.

  1. Check your local file is reachable Verify datapill can open your file before running a full ingest.
    bash
    $ dp ingest check local --path /data/exports/orders.parquet
  ✔ connected  latency 1ms
  2. Ingest the file Run a full ingest and materialize the result as a versioned Parquet artifact.
    bash
    $ dp ingest run local \
    --path        /data/exports/orders.parquet \
    --materialize \
    --schema

  run_id   a1b2c3d4
  ref      local:a1b2c3d4

  column          type
  ─────────────── ─────────
  id              Int64
  customer_id     Int64
  total           Float64
  created_at      Datetime

  artifact  .datapill/artifacts/local/a1b2c3d4.parquet
  3. Profile the artifact Run a statistical profile to detect column types, nulls, correlations, and data-quality warnings.
    bash
    $ dp profile run a1b2c3d4 --mode full

  run_id   b2c3d4e5
  ref      profile:b2c3d4e5
  4. Inspect the profile View dataset statistics, column summaries, correlations, and warnings.
    bash
    $ dp profile show b2c3d4e5

▸ DATASET
  run          b2c3d4e5
  parent       a1b2c3d4
  rows         124,532
  columns      4
  null %       0.42%
  duplicate %  0.01%
  5. Check warnings Review any data-quality issues flagged during profiling.
    bash
    $ dp profile warnings b2c3d4e5 --severity warn

  [WARN ] total                   HIGH_SKEW
  [WARN ] (dataset)               DUPLICATE_ROWS
  6. Inspect the ingest artifact View metadata for the ingestion run.
    bash
    $ dp artifact show a1b2c3d4

  run          a1b2c3d4
  pipeline     local
  timestamp    2026-05-02 09:14
  sample       no
  materialized yes
  path         artifacts/local/a1b2c3d4.parquet

datapill ingest

The ingest command group handles running and validating data ingestion pipelines.

ingest run

Read data from a source and optionally store it as a versioned artifact.

synopsis
dp ingest run SOURCE --config FILE [OPTIONS]

Options

Argument / FlagTypeDefaultDescription
SOURCE requiredstring…source type: postgres, mysql, sqlite, s3, local, kafka, rest
--config / -c optionalstring-path to config .json file (required for postgres, mysql, sqlite, kafka, rest, s3; optional for local)
--table / -t optionalstring-table name (postgres, mysql, sqlite)
--query / -q optionalstring-raw SQL query (postgres, mysql, sqlite)
--topic optionalstring-kafka topic
--path / -p optionalstring-file path (local: full path to file; s3: key within bucket)
--endpoint / -e optionalstring-REST endpoint path
--params optionalstring-path to params .json file
--sample optionalflagfalseread a sample instead of full data
--sample-size optionalint10000number of rows to sample
--batch-size optionalint-rows per streaming batch
--materialize / -m optionalflagfalsewrite output to parquet artifact
--store optionalstring.datapillartifact store directory
--schema optionalflagfalseprint column schema after ingest
--mkdir optionalflagfalsecreate base_path directory if it does not exist (local source only)

Examples

bash
$ dp ingest run postgres \
    --config  pg.json \
    --query   "SELECT id, email FROM users WHERE active = true" \
    --materialize \
    --schema
bash
$ dp ingest run mysql \
    --config mysql.json \
    --table  products
bash
$ dp ingest run postgres \
    --config      pg.json \
    --table       events \
    --sample \
    --sample-size 5000 \
    --schema
# Useful for validating schema before reading the full table
bash
$ dp ingest run s3 \
    --config s3.json \
    --path   "data/2026/orders.parquet" \
    --materialize

artifact show

Display full metadata for a single artifact by run ID.

synopsis

artifact lineage

Trace the ancestry chain of an artifact back to its root. Useful when pipelines are chained.

bash
$ dp artifact lineage a1b2c3d4

raw_ingest   0000ffff  2026-04-29 08:00
  └─ clean   1234abcd  2026-04-29 08:15
    └─ orders   a1b2c3d4  2026-05-02 09:14

artifact delete

Delete a single artifact and its associated files on disk. Prompts for confirmation unless --yes is passed.

If the artifact has child artifacts (e.g. a profile derived from an ingest), the delete will fail with a list of blocking children. Use --cascade to delete the entire subtree in leaf-to-root order.

synopsis
dp artifact delete RUN_ID [--yes] [--cascade] [--store PATH]

Options

FlagTypeDefaultDescription
RUN_ID requiredstring…run id of the artifact to delete
--yes / -y optionalflagfalseskip confirmation prompt
--cascade optionalflagfalsealso delete all child artifacts (leaf → root order)
--store optionalstring.datapillartifact store directory

Examples

bash
# delete a leaf artifact
$ dp artifact delete a1b2c3d4 --yes
  [OK] deleted  a1b2c3d4

# delete fails when children exist
$ dp artifact delete 0000ffff --yes
  [FAIL] cannot delete 0000ffff: has 2 child artifact(s): 1111aaaa, 2222bbbb

# cascade — shows subtree preview, then deletes leaf → root
$ dp artifact delete 0000ffff --cascade
  will delete 3 artifact(s):
    2222bbbb  profile
    1111aaaa  profile
    0000ffff  ingest  materialized
confirm? [y/N]: y
  [OK] deleted  0000ffff
⚠️ Cascade is irreversible
--cascade deletes the artifact and every descendant in the lineage tree, including their materialized files on disk. There is no undo.

artifact purge

Bulk-delete root artifacts and their entire subtrees, keeping the N most recent roots. --keep counts root artifacts (ingests with no parent), not individual artifact records — so --keep 3 always preserves 3 complete lineage trees.

synopsis
dp artifact purge [--pipeline NAME] [--keep N] [--samples-only] [--yes] [--store PATH]

Options

FlagTypeDefaultDescription
--pipeline / -p optionalstring-limit to a specific pipeline name
--keep / -k optionalint0number of most recent root artifacts to keep
--samples-only optionalflagfalseonly delete sample artifacts within each subtree
--yes / -y optionalflagfalseskip confirmation prompt
--store optionalstring.datapillartifact store directory

Examples

bash
# keep the 3 most recent root artifacts, purge everything older
$ dp artifact purge --keep 3 --yes
  [OK] purged  9 artifact(s)

# purge all roots for a specific pipeline (keeps none)
$ dp artifact purge --pipeline ingest --yes
  [OK] purged  7 artifact(s)

# purge only sample runs, keep 5 recent roots
$ dp artifact purge --keep 5 --samples-only --yes
  [OK] purged  3 artifact(s)
ℹ️ Subtree semantics
Each root that falls outside --keep is deleted together with all its descendants (profiles, preprocessed runs, etc.) in leaf-to-root order. A root with a child that is inside the keep window will not be deleted.

artifact usage

Show total artifact count and disk usage for the store.

bash
$ dp artifact usage

  artifacts    42
  disk usage   1.3 GB

datapill preprocess

Apply transformation operations to an existing ingest or preprocess artifact to produce a cleaned, feature-engineered dataset. Results are tracked as child artifacts linked to their parent via lineage.

ℹ️ Lazy by default
Without --materialize, preprocess run records the op config in the artifact store but does not write any data to disk. Call preprocess materialize later to apply and persist the ops when you need the output.

preprocess run

Apply one or more transformation ops to a parent artifact. Ops can be passed inline with --op or supplied as a JSON file with --ops.

synopsis
dp preprocess run PARENT_RUN_ID [--op OP]... [--ops FILE] [OPTIONS]

Options

Argument / FlagTypeDefaultDescription
PARENT_RUN_ID requiredstring…run id of the parent artifact (ingest or preprocess)
--op optionalstring (repeatable)-inline op in format group.type:key=value:key=value
--ops optionalstring-path to a JSON file containing an array of op objects
--materialize / -m optionalflagfalseapply ops and write output to parquet artifact
--schema optionalflagfalseprint column schema after run
--store optionalstring.datapillartifact store directory

Examples

bash
# apply a pre-written ops file to a parent ingest artifact
$ dp preprocess run 0f34bcc3     --ops tests/fixtures/ops/clean_employees.json     --schema
[OK] 150,750 rows 21 cols
⠹ preprocess complete 0:00:00
- RUN
  run:  146e9c2a
  ref:  preprocess:146e9c2a
bash
# chain multiple inline ops
$ dp preprocess run 0f34bcc3     --op "clean.drop_null:cols=email,salary"     --op "parse.lower:cols=email"     --op "transform.log_transform:cols=salary:base=log10"
bash
# run and write parquet in one step
$ dp preprocess run 0f34bcc3     --ops        clean_employees.json     --materialize     --schema
- OUTPUT
  parquet: artifacts\c22b374a\data.parquet
⚠️ --op and --ops are mutually exclusive
Use one or the other per invocation. Mixing them will produce an error.

preprocess materialize

Replay and persist the ops from an existing (non-materialized) preprocess artifact, writing the result to a parquet file. Creates a new artifact entry linked to the same parent.

synopsis
dp preprocess materialize RUN_ID [--schema] [--store PATH]

Options

Argument / FlagTypeDefaultDescription
RUN_ID requiredstring…run id of the existing preprocess artifact to materialize
--schema optionalflagfalseprint column schema after materializing
--store optionalstring.datapillartifact store directory
bash
$ dp preprocess materialize 146e9c2a --schema
[OK] 150,750 rows 21 cols
⠹ preprocess complete 0:00:00
- RUN
  run:  c22b374a
  ref:  preprocess:c22b374a
- OUTPUT
  parquet: artifacts\c22b374a\data.parquet
ℹ️ Already materialized
If the artifact is already materialized, the command prints the existing path and exits without re-running ops.

Op format

Inline ops follow the pattern group.type:key=value:key=value. Multi-value params use comma separation:

bash
# drop two columns
$ dp preprocess run 0f34bcc3 --op "schema.drop_columns:cols=phone,tags"

# bin age into buckets
$ dp preprocess run 0f34bcc3 --op "transform.bin:col=age:breaks=0,25,35,45,55,100"

# one-hot encode a categorical column
$ dp preprocess run 0f34bcc3 --op "transform.encode:col=dept:method=onehot"

Ops JSON file format

A JSON ops file is an array of op objects — the same keys used in inline format, but as structured JSON:

clean_employees.json
[
  { "group": "clean",     "type": "drop_null",     "cols": ["email", "salary"] },
  { "group": "clean",     "type": "clip",          "cols": ["age"], "min": 16, "max": 70 },
  { "group": "clean",     "type": "fill_null",     "cols": ["bonus_pct"], "value": 0 },
  { "group": "schema",    "type": "drop_columns",  "cols": ["tags", "phone"] },
  { "group": "parse",     "type": "lower",          "cols": ["email"] },
  { "group": "transform", "type": "log_transform", "cols": ["salary"], "base": "log10" },
  { "group": "reshape",   "type": "dedup",          "cols": ["email"], "keep": "first" }
]

datapill export

Export any artifact produced by ingest, preprocess, or profile to a file. Supports six output formats. All three required inputs can be prompted interactively when omitted.

export run

Export an artifact to a file. RUN_ID, --format, and --output are optional — omitting any of them triggers an interactive prompt.

synopsis
dp export run [RUN_ID] [--format FMT] [--output PATH] [options]

Options

Argument / FlagTypeDefaultDescription
RUN_ID optionalstring—run id of the artifact to export (ingest, preprocess, or profile). Omit to pick interactively.
--format / -f optionalstring—output format: csv parquet json jsonl excel html
--output / -o optionalpathautooutput file or directory. If omitted, a filename is generated automatically.
--delimiter / -d optionalchar,CSV column delimiter
--compression optionalstringsnappyParquet compression: snappy | gzip | zstd | lz4 | uncompressed
--partition-by optionalstring—Parquet partition columns, comma-separated (e.g. dept,year)
--sheet optionalstringSheet1Excel sheet name
--filename optionalstringautooutput filename when --output is a directory
--interactive / -i optionalflagonprompt for missing values. Use --no-interactive / -I to disable for CI.
--store optionalpath.datapillartifact store directory

Examples

bash
# fully interactive — pick artifact, format, and output path from prompts
$ dp export run

# ingest → CSV
$ dp export run 54bede20 --format csv --output ./employees.csv

# ingest → Parquet, gzip compression
$ dp export run 54bede20 --format parquet --output ./out.parquet --compression gzip

# ingest → partitioned Parquet (output must be a directory)
$ dp export run 54bede20 --format parquet --output ./out/ --partition-by dept,year

# ingest → Excel
$ dp export run 54bede20 --format excel --output ./employees.xlsx --sheet Employees

# profile → HTML report
$ dp export run 47689b73 --format html --output ./report.html

# auto output path — resolves to ./ingest_54bede20.csv
$ dp export run 54bede20 --format csv --no-interactive

  output:  ./ingest_54bede20.csv

# example output after a successful export
$ dp export run 54bede20 --format parquet --output ./out.parquet

── output ───────────────────────────────────────────
  path:    ./out.parquet
  size:    4.2 MB
  format:  parquet
  options: compression=snappy
ℹ️ Format restrictions by artifact type
profile artifacts only support json and html. ingest and preprocess artifacts support csv, parquet, json, jsonl, and excel. Passing the wrong format produces a clear validation error before any data is read.
💡 Extension mismatch
If --output has an extension that does not match --format, the CLI will report the error and suggest the correct path: → try: --output ./data.csv

Streaming Large Datasets

By default, datapill loads an entire result set into memory. For large tables, add --batch-size to enable chunked streaming:

bash
$ dp ingest run postgres \
    --config     pg.json \
    --table      events \
    --batch-size 100000 \
    --materialize
⚠️ Memory note
Even with streaming, each batch is held in memory until written. Set --batch-size low enough to fit comfortably within your available RAM.

Recommended batch sizes by source:

SourceRecommended batch-sizeNotes
PostgreSQL100 000 – 500 000Server-side cursor via asyncpg
MySQL50 000 – 200 000SSCursor for row-by-row fetch
SQLite50 000DuckDB Arrow reader handles chunking
S3 (CSV / JSON)8 MB chunksControlled by chunk_size in config
Kafka500 messagesControlled by max_poll_records in config

Working with Artifacts

Every ingest run creates an artifact entry in .datapill/store.db. With --materialize, the data is also written to .datapill/artifacts/<pipeline>/<run_id>.parquet.

Reading a materialized artifact with DuckDB

Recommended workflow

  • Use --sample first to validate schema and data quality cheaply.
  • Run a full ingest with --materialize once the sample looks correct.
  • Use artifact lineage to trace how a derived dataset was produced.
  • Use preprocess run without --materialize to record a transformation plan cheaply, then call preprocess materialize only when you need the parquet output.
  • Chain multiple preprocess artifacts (each pointing to the previous as parent) to build multi-step pipelines tracked in full lineage.
  • Schedule artifact purge --keep 5 in CI to prevent unbounded disk growth.
  • Use artifact delete --cascade to remove an entire lineage tree in one command.
  • Delete artifacts leaf-first when not using --cascade — deleting a parent while children exist will fail.
